# Spree 医药 B2B 订货系统设计

## 背景

本项目面向药店采购场景：药店在平台下单购买药品，平台基于全国货盘完成报价、库存匹配、订单履约和配送。系统选择 Spree 作为电商交易底座，但不把医药行业的货盘、批号、效期、资质、配送和追溯逻辑硬塞进 Spree 原始商品模型，而是在 Spree 旁边新增一套医药 B2B 扩展域。

## 设计目标

- 药店可以完成注册、资质审核、搜索药品、下单、查看配送和票据资料。
- 平台可以维护药品主数据、货盘方、报价、库存、批号效期、区域可售和订单履约。
- 一个药品可以关联多个货盘方、多个仓库、多个批号、多个效期、多个价格。
- 前台是否展示具体货盘方可以通过后台策略控制。
- 第一版优先跑通平台统一运营的采购闭环，后续再开放供应商后台和自动化履约。

## 总体架构

```text
药店端
  药品搜索
  商品详情
  购物车
  下单
  订单跟踪
  票据/随货资料

Spree 电商核心
  Product / Variant
  Cart / Order / LineItem
  StockLocation / Shipment
  Payment / Adjustment
  Admin / API

医药 B2B 扩展
  药店资质
  供应商/货盘方
  药品主数据
  供应商报价
  批号效期库存
  区域可售规则
  订单分配/拆单
  发票对账
  追溯码/随货资料
```

Spree 负责通用电商交易能力；医药 B2B 扩展负责行业特有的合规、货盘和履约规则。

## 核心建模原则

不要设计成：

```text
一个药品 = 一个商品 = 一个库存 = 一个价格
```

应该设计成：

```text
一个药品主数据
  -> 一个或多个 Spree Variant
  -> 多个供应商报价
  -> 多个仓库库存
  -> 多个批号/效期
  -> 多个区域可售规则
```

前台可购买的是 Spree Variant；真实供货能力来自供应商报价和批号效期库存。

## 保留的 Spree 模型

- `spree_products`：商品展示层，承载药品销售页。
- `spree_variants`：最小销售规格，与药品主数据绑定。
- `spree_orders`：药店主订单。
- `spree_line_items`：订单商品行。
- `spree_stock_locations`：仓库或发货点，可与供应商仓库关联。
- `spree_shipments`：发货单。
- `spree_payments`：支付记录。
- `spree_adjustments`：价格调整、运费、优惠等。

## 新增医药 B2B 模型

### 药店与资质

- `pharmacies`
  - 药店主体信息、联系人、区域、经营状态。
- `pharmacy_users`
  - 药店员工账号、角色、权限。
- `pharmacy_licenses`
  - 营业执照、药品经营许可证、GSP 信息、有效期、审核状态。

### 供应商与货盘方

- `suppliers`
  - 货盘方主体、公司名称、结算方式、服务区域、运营状态。
- `supplier_licenses`
  - 供应商资质、经营范围、证照有效期、审核状态。
- `supplier_warehouses`
  - 供应商仓库、发货区域、配送能力、温控能力。

### 药品主数据

- `drug_master`
  - 通用名、商品名、规格、剂型、厂家、批准文号、包装单位、是否处方药、储存条件、温控要求。
- `drug_variant_links`
  - `drug_master` 与 `spree_variants` 的绑定关系。

### 货盘与库存

- `supplier_offers`
  - 供应商、药品规格、报价、起订量、限购量、报价有效期、状态。
- `supplier_offer_regions`
  - 报价可售省市区、配送时效、运费规则。
- `drug_batch_stocks`
  - 供应商、仓库、药品规格、批号、效期、可用库存、锁定库存、库存状态。

### 订单履约

- `order_allocations`
  - 每个订单行被分配到哪个供应商、仓库、报价、批号、数量和成交价格。
- `supplier_fulfillments`
  - 按供应商/仓库拆出的履约单，记录拣货、发货、配送、签收。
- `fulfillment_documents`
  - 随货同行单、质检资料、冷链记录、配送凭证。

### 财务与追溯

- `invoice_records`
  - 发票抬头、发票状态、开票主体、发票文件、对账状态。
- `settlement_records`
  - 平台与供应商、平台与药店之间的结算记录。
- `traceability_records`
  - 追溯码、批号、订单、发货单、上传状态。

## 供应商展示策略

前台是否显示供应商由后台配置控制：

```text
supplier_visibility_mode
```

可选值：

- `hidden`：隐藏供应商，药店只看到平台价格、库存、预计到货和平台服务标签。
- `partial`：部分展示，例如显示发货仓、区域仓、平台优选、配送时效，不显示具体公司名。
- `visible`：完全展示供应商名称、报价、起订量、效期、库存、配送时效。

第一版默认使用 `hidden`。这样平台统一面对药店，避免药店绕过平台直接联系货盘方，也降低前台信息复杂度。

无论前台是否隐藏供应商，后台订单必须完整记录：

```text
supplier_id
supplier_name_snapshot
offer_id
warehouse_id
batch_no
expiry_date
allocated_price
allocated_quantity
```

这些字段用于履约、售后、发票、对账和药品追溯。

## 下单与履约流程

```text
药店登录
  -> 检查药店资质状态
  -> 搜索药品
  -> 系统筛选可售报价与库存
  -> 按展示策略返回价格、库存、时效和供应商信息
  -> 药店加入购物车
  -> 下单时锁定报价与库存
  -> 创建 Spree 主订单
  -> 创建订单分配记录
  -> 按供应商/仓库/批号生成履约单
  -> 供应商或平台发货
  -> 上传随货资料、发票、追溯码
  -> 药店签收
  -> 平台完成对账结算
```

第一版可以先支持半自动分配：系统给出推荐货盘，运营人员确认后生成履约单。后续再升级为自动分配。

## 货盘匹配规则

先过滤不可用货盘：

- 药店资质未通过，不允许下单。
- 供应商资质过期，不允许分配。
- 药品未上架或不可售，不允许展示。
- 药店所在区域不在报价可售范围内，不允许展示。
- 可用库存不足，不允许下单。
- 效期低于平台设定阈值，不允许分配。

再排序可用货盘：

- 价格更低优先。
- 效期更远优先。
- 配送时效更快优先。
- 库存更稳定优先。
- 平台设定供应商优先级更高优先。

## MVP 范围

第一版聚焦平台统一运营，不开放供应商自助后台。

必须包含：

- 药店注册、登录、资质上传、后台审核。
- 药品主数据管理。
- 供应商和供应商资质管理。
- 货盘 Excel 导入。
- 批号效期库存管理。
- 药店端搜索、商品详情、购物车、下单。
- 后台半自动货盘分配。
- 发货、签收、订单状态流转。
- 基础发票记录和对账记录。
- 供应商前台展示策略，默认隐藏。

暂不包含：

- 供应商自助后台。
- 自动比价和全自动拆单。
- 供应商 API 实时库存同步。
- 账期授信。
- 电子发票深度集成。
- 冷链 IoT 设备接入。
- 追溯码自动上报监管平台。

## 后续演进

第二阶段：

- 开放供应商后台。
- 支持供应商自主维护货盘、库存、发货和票据。
- 支持 `partial` 和 `visible` 供应商展示模式。
- 支持自动拆单和自动分配。

第三阶段：

- 对接供应商 ERP/WMS。
- 增加药店账期、授信和风控。
- 增加电子发票、冷链记录和追溯码自动化。
- 增加平台与供应商结算中心。

## 风险与约束

- Spree 不是中国医药 B2B 系统，需要新增大量行业模型。
- Ruby/Rails 技术栈需要团队具备维护能力。
- 药品资质、经营范围、批号效期、随货资料和追溯记录必须前置设计，不能后补。
- 第一版必须控制范围，先跑通交易和履约闭环，再做供应商开放平台。

## 当前决策

- 使用 Spree 作为电商交易底座。
- 货盘方作为供应商/履约主体建模，不直接等同于 Spree 商品所属商家。
- 供应商前台展示支持隐藏、部分展示、完全展示三种模式。
- 第一版默认隐藏供应商。
- 第一版以平台统一运营货盘为主，不开放供应商自助后台。
